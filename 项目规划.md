# 🎯 项目二：AI 客服/销售智能助手（Multi-Agent 系统）— 项目规划

## 一、项目定位

> 做一个基于 Multi-Agent 架构的智能客服/销售助手，
> 用户提出问题 → 主 Agent 判断意图 → 路由到子 Agent → 调用工具 → 回复用户

### 对标 FDE 岗位的能力点
| 岗位要求 | 本项目体现 |
|---------|-----------|
| AI 应用项目经验 | 完整的多 Agent 应用 |
| 会 AI 编程工具（vibe coding） | 用 Cursor 等工具开发 |
| 深度使用 LLM | 调用 LLM 做意图识别、应答生成 |
| Agent 框架（LangChain/LlamaIndex） | 基于 LangGraph 实现 |
| 云平台部署 | 可选：Docker + 云部署 |

### 与项目一的互补
- 项目一：RAG（检索增强生成）→ **"理解"**
- 项目二：Multi-Agent（多智能体协作）→ **"行动"**
- 面试时可以串起来讲：**"从理解知识到执行任务"**

## 二、技术选型

| 组件 | 技术 | 原因 |
|------|------|------|
| 编程语言 | Python 3.11+ | AI 生态最完善 |
| Agent 框架 | **LangGraph** | 比 LangChain Agent 更适合多 Agent 编排，有图状态机 |
| LLM | OpenAI API（首选）+ Ollama（本地备选） | 开发期用 OpenAI 快，演示时可用本地模型 |
| 向量数据库 | ChromaDB（复用项目一） | 已熟悉，轻量 |
| 后端 | FastAPI | 简单高效，容易部署 |
| 前端 | Streamlit | 快速出界面，面试演示够用 |
| 工具调用 | 内置函数（查库存/写工单） | 模拟真实场景，不依赖外部服务 |

### LangGraph 选型理由
- 比 LangChain Agent 更适合多 Agent 场景：有状态图、分支、循环
- 面试中 LangGraph 比普通 Agent 更有技术深度
- 2024-2025 年的热门框架，面试常见话题

## 三、需求分析

### 3.1 业务场景

模拟一个**电子产品电商的客服/销售助手**，支持三类场景：

| 场景 | 示例 | 处理 Agent |
|------|------|-----------|
| **售前咨询** | "推荐一款适合办公的笔记本"<br>"iPhone 15 性价比怎么样？" | 销售 Agent |
| **售后服务** | "我的订单还没到"<br>"怎么退货？" | 客服 Agent |
| **技术支持** | "电脑蓝屏怎么办？"<br>"WiFi 连不上" | 技术 Agent |
| **闲聊/模糊** | "你好" / "你是谁" | 主 Agent 直接处理 |

### 3.2 功能需求

1. **意图识别**：主 Agent 判断用户问题属于哪一类
2. **Agent 路由**：分发到对应子 Agent
3. **工具调用**：各 Agent 可调用预设工具
4. **记忆管理**：对话历史保留
5. **Fallback**：识别失败时兜底回复
6. **UI 界面**：Streamlit 聊天界面

### 3.3 非功能需求

- 响应时间 < 10s（本地 LLM 可接受）
- 错误处理：各环节有 try-catch
- 可扩展：新增 Agent 只需加一个节点

## 四、系统架构

```
用户输入
    │
    ▼
┌──────────────┐
│  主 Agent    │ ← 意图识别（LLM 分类）
│  (Supervisor)│
└──────┬───────┘
       │
       ├─ 售前 → ┌──────────────┐
       │         │ 销售 Agent    │ → 工具：查库存/推荐商品/算价格
       │         └──────────────┘
       ├─ 售后 → ┌──────────────┐
       │         │ 客服 Agent    │ → 工具：查订单/退货流程/投诉
       │         └──────────────┘
       ├─ 技术 → ┌──────────────┐
       │         │ 技术 Agent    │ → 工具：查手册/诊断/常见问题
       │         └──────────────┘
       └─ 闲聊 → 主 Agent 直接回复
```

## 五、数据结构设计

### 5.1 消息结构

```python
class Message(BaseModel):
    role: str          # "user" | "assistant" | "system"
    content: str       # 消息内容
    agent: str | None  # 处理该消息的 Agent 名称

class ChatState(BaseModel):
    messages: list[Message]      # 对话历史
    current_agent: str | None    # 当前处理 Agent
    intent: str | None           # 识别到的意图
    tool_results: list[str]      # 工具调用结果
    error: str | None            # 错误信息
```

## 八、风险与备选方案

| 风险 | 影响 | 备选方案 |
|------|------|---------|
| OpenAI API 需付费 | 开发调试需网络和费用 | 先 Mock LLM 调用，后期再接入真实 API |
| LangGraph 学习曲线 | 如果上面不顺手可能开发慢 | 降级为 LangChain Agent Executor，功能略简单但够用 |
| 本地 LLM 效果差 | 意图识别不准 | 先调好 Prompt，用 OpenAI 稳定后考虑本地 |
| 时间不够做全 | 不能按期完成 | 优先核心链路，省略 UI 美化/日志等次要功能 |
